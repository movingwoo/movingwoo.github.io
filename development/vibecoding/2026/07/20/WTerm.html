    프라이빗 웹 터미널

    개요

    AI 세션 여기저기 만들고 다니니 관리가 귀찮다.
    개발 환경도 여기저기 흩뿌려놨더니 매번 ssh 접속하는 것도 짜증난다.
    주요 개발환경에 백그라운드로 띄어두고 웹으로 접근할 수 있는 개인용 웹 터미널을 만들자.

    설계

    FastAPI를 사용한다.
    예전에 잠깐 공부할때 상당히 매력적이라고 느꼈는데 이번에 써야지.

    xterm.js

    브라우저에서 터미널을 사용하려면 xterm.js라는 훌륭한 라이브러리가 존재한다.
    서버에는 PTY를 열고 PTY와 웹소켓 아이에서 데이터를 중계한다.

    현재 클로드 코드를 적극적으로 사용하고 있어서 클로드 코드 기준으로 설계한다.
    다만 언제 코덱스 등 다른 AI로 넘어갈지 모르니 확장을 염두에 두자.
    클로드 코드 세션 기록은 .claude/projects/에 존재하므로 해당 경로에 의존한다.

    기타 기능으로 셸 세션을 위해 bash -l을 띄우는 기능도 마련한다.
    아무래도 민감 데이터에 웹으로 접근하는 것이기 때문에 1차 방어기능 마련이 필수이며
    자체적으로 2차 인증을 위해 비밀번호 입력 기능을 간단하게 넣어둔다.

    구현

    projects.json                       # 프로젝트 설정
server/
├── config.py                       # projects.json 로더
├── session.py                      # PTY 세션 수명 주기
└── main.py                         # FastAPI 메인 앱
static/
├── index.html, app.js, style.css   # xterm.js UI
└── static/vendor/                  # xterm.js 로컬 사본 (수정 X)

    {
  "host": "127.0.0.1",
  "port": 8877,
  "grace_seconds": 60,
  "projects": [
    {
      "name": "example-project",
      "path": "/home/user/example-project"
    },
    {
      "name": "remote-project",
      "path": "/home/user/remote-project",
      "ssh": "user@remote-host"
    }
  ],
  "password_sha256": "패스워드 hash"
}

    호스트 바인딩은 웬만하면 127.0.0.1만 해두는게 안전할 것이다.
    0.0.0.0으로 해두었을 때의 참사는 생각하기도 싫다.

    패스워드의 경우 password_sha256에 등록해둔 hash 값으로 인증을 진행한다.
    단순 2차인증 수단일 뿐이니 앞단에 충분한 보안 대책이 필요하다.

    caddy 연동을 위한 위한 뻘짓

    본 터미널이 올라갈 서버의 caddy 웹서버는 docker로 올라가 있다.
    여기서 발생하는 문제, 호스트 네트워크가 도커와 달라 단순 기동 시 리버스 프록시 설정이 불가능하다.
    caddy에 호스트 네트워크를 붙여주면 되는데 이러면 포트를 열어줘야하는 상황.
    안전한 IT 라이프를 위해 인바운드 허용은 자제하는 편이 낫다.
    따라서 유닉스 소켓 리스닝 방식을 사용한다.

    {
  "uds": "/path/run/wterm.sock",
}

    TCP 바인딩 대신 uds에 설정한 소켓으로 리슨한다.
    이 소켓 파일을 caddy 컨테이너에 마운트하면 인바운드 포트를 신경쓰지 않아도 된다.

    services:
  caddy:
    volumes:
      - /path/run:/run/wterm:ro

    소켓 폴더를 caddy에 마운트 하고 리버스 프록시에는 reverse_proxy unix//run/wterm/wterm.sock 형식으로 소켓을 지정한다.

    프로토타입 완성

    테스트 중 가슴아픈 버그가 발생했다.
    mac 환경에서 Safari/Webkit 한글 IME 조합 버그가 존재한다.
    윈도우 및 타 브라우저 환경에서는 정상 동작한다.
    mac 환경에서 한글 이상해지는거 하루이틀 문제가 아니긴한데…
    크롬을 애용하자.
